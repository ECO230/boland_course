GROUP PROJECT – DATA ANALYSIS PROJECT

Project Prompt

Overview

Note

OVERVIEW

The purpose of this project is to introduce you to analysis of organizational data using descriptive statistics, data visualization, and inferential statistics. While completing this project, you will develop and practice the skills necessary to communicate findings. Communication will occur through an oral presentation, professionally prepared presentation, and documentation of your work in RScripts, RMarkdown or other formats as required by your instructor.

Assignment learning objectives

Note

ASSIGNMENT LEARNING OBJECTIVES MAPPED TO COURSE LEARNING OBJECTIVES

Assignment Learning Objective Course Learning Objectives
1. Apply scripting skills (in RStudio) to prepare visual and descriptive summaries of workforce aging and use inferential statistics (where appropriate) to address the case prompt and inform conclusions. 2. Describe, summarize, and interpret data using descriptive statistics, inferential statistics, and data visualization.<br>3. Develop foundational skills related to spreadsheets, interactive data visualization software, and scripting languages used for data analysis and visualization.<br>4. Apply research design and data analysis best practices to complete case-based projects and to critique others’ analyses.
2. Organize main findings, clearly support conclusions with analysis, and translate in a professional business presentation to a nontechnical audience. 5. Communicate the purpose, methods, and results of analysis to authentic audiences in appropriate written and oral formats.
3. Prepare thorough documentation of work (e.g., in annotated RScripts or RMarkdown) for a technical audience following best practices for organizing and annotating code.

Prompt

Note

PROMPT

By this point you should have a pretty good understanding of what type of data is in your dataset and you should have some preliminary ideas of stories you might be able to tell with this data set. In this project you will come up with an analysis plan, develop hypotheses you want to test, clean up your data set and test your hypotheses statistically. You will also prepare descriptive and visual analysis of your data and prepare a formal presentation to summarize your key findings.

Resources

Note

RESOURCES

Warning

RECOMMENDATIONS FOR YOUR ANALYSIS

  1. Decide what stories you will be working on as soon as possible. Your story may evolve over time, but having an idea of what information you are trying to get out of your data set will assist in cleaning up your data set, defining and testing your hypotheses and developing your presentation.
  2. Have a discussion about how you want to divide work within your group. Are you each going to be working on one story, are some members going to specialize in data analysis while others specialize on developing the presentation.
  3. The backbone of your analysis will be performed and documented in R. This will include cleaning your data set to prepare it for analysis. Descriptive and inferential statistics and, if you prefer, visualizations. You can export data and perform analysis in Excel/Tableau, but the data set you are using for analysis should be prepared in R. This not only makes it easier to document how your data was prepared, but also helps to ensure that everyone in your group is working with the same data set.
  4. With the exception of the inferential statistics (which we will start using in Week 6) you have already performed some version of the analysis you are expected to do for this project in earlier assignments. This is the same type of work we have been doing so far in the class, just more of it. Feel free to recycle analysis that you did for previous assignments; if it supports your story or helps to address your hypothesis there is no need to start over from scratch.
  5. Make good use of the multiple weeks you have to work on this project; you do not want to wait until the last minute.
  6. Pick stories that you think are interesting. It will make your presentation more compelling if you are actually invested in the story you are telling. It will also make your analysis seem like less of a chore.
Warning

DATA

Full data sets are provided in the project templates per topic. Some of these data sets are quite large so a good early step is to filter out data that is not needed to address your hypotheses. It will make your analysis run faster and make the sizes of your files easier to deal with if you are doing analyses in Excel or Tableau. Some examples of common data cleaning tasks are included in your posit.cloud project; feel free to adapt them to your needs.

Warning

PRESENTATION MATERIALS

Your presentation may be prepared in either Google Slides or PowerPoint. If using PowerPoint, you are encouraged to use Office Online to enable all team members to access the presentation and contribute. Effective slide design will be covered in class, and the principles of effective design are reflected in the rubric included under Grading Criteria. All team members are responsible for the content of all slides and the presentation should be cohesive, even if you have delegated the design of certain sections to separate team members.

Warning

ORAL PRESENTATION

All teams will present their work in a formal business presentation. Ask your instructor about the presentation location, times, and venue (e.g., seated conference room style vs. standing). The following expectations apply to all sections:

  1. No notecards
  2. Dress is business casual or more formal
  3. All team members must speak
  4. The audience may interrupt to ask questions during the presentation. Plan for 8–10 minutes of content, not including questions.
  5. Presentation content should be non-technical. Slides should not contain technical output of hypothesis tests and the oral delivery should not include detailed discussions of statistical tests. However, students should be prepared to answer technical questions if prompted and may choose to have supplemental content prepared at the end of the presentation materials in anticipation of technical questions.
Warning

DOCUMENTING YOUR ANALYSIS

All teams must create RScripts or RMarkdown documents containing sufficient annotations to enable an outside reader to understand and replicate their work. Code should be “tidy”, meaning code following each pipe in a dplyr call begins on a new line. For ease of dividing work across team members, you may prepare separate RScripts or RMarkdown documents for each piece of your analysis. If you choose to do this, provide a single document that lists the file name of each RScript or RMarkdown with a brief description of contents. All team members are responsible for the content of all documents submitted. If one is unsatisfactory, the entire team’s grade will suffer.

Formal documentation of work

Warning

FORMAL DOCUMENTATION OF WORK

The following simple rubric will be used to assign grades for documentation of work. Please refer to any supplemental guidelines provided by your instructor.

  1. Is documentation sufficient to follow the analysis conducted? Can a data analyst follow your process from raw data to output/conclusions?
  2. Does code run without errors from start to finish when beginning with an empty environment?
  3. Does code reproduce all content included in the presentation materials without major omissions?
  4. Are analyses performed in Excel/Tableau sufficiently documented? Can a data analyst recreate your analysis with the documentation provided?
  5. Is organization across and within files effective for matching presented work to the code that produced it?
  6. Are there discrepancies between the actions the code performs and the way the output is presented in the slides or oral presentation? For example, does the code use filters appropriately to produce analysis for a subset of workers as reported in the slides?
  7. Does the code contain appropriate inferential statistics to accompany descriptive analysis presented in the presentation?
